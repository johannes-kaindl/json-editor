# 1.3.0 — JSON Schema Validation — Design-Spec

**Projekt:** `obsidian-json-editor`
**Datum:** 2026-05-27
**Status:** Spec, freigegeben (autonomer Run nach 1.2.0)
**Entstanden aus:** Backlog-Item §4 in `AGENTS.md` / 1.0.0-Scope-Decomposition.

1.3.0 fügt **JSON Schema validation** hinzu. Wenn der Tree-/Source-Mode-Editor weiß, *was* ein Datei sein soll, kann er Fehler in Echtzeit anzeigen statt sie der Save-Zeit oder der nächsten manuellen Inspektion zu überlassen.

---

## 1. Scope

**IN:**
- `ajv@8` als runtime-Dependency (in `dependencies`, nicht devDeps)
- Reines Core-Modul `src/core/schema.ts` — `compileSchema(text)` + `validate(value)` ohne DOM
- **Companion-File-Pattern**: Datei `foo.json` → Plugin sucht `foo.schema.json` im selben Ordner
- Validation läuft bei jedem Change in beiden Modi
- **Schema-Banner** oberhalb der Toolbar zeigt Anzahl Fehler an
- **Inline-Markierung** auf Tree-Rows: rote Border + Tooltip mit Fehlermeldung
- Setting `validateAgainstSchema: boolean` (default true) — Master-Switch
- Setting `companionSchemaSuffix: string` (default `.schema.json`)

**OUT:**
- `$schema` URL-Fetching (Cross-Origin / Vault-Path-Resolution → too many edge cases)
- Per-Workspace oder Settings-pinned Schema (Schema MUSS als Companion-File neben der Datei liegen)
- Schema-Generierung aus Beispieldaten
- Sub-Schema-Drilldown UI

## 2. Architektur

```
src/core/schema.ts                 pure, Ajv-wrapped
  compileSchema(text) → CompiledSchema | { error }
  validate(value, schema) → ValidationResult { ok, errors: PathError[] }
  PathError = { path: JsonPath, message: string }

src/obsidian/SchemaBanner.ts       UI-Komponente
  setErrors(count: number)
  setVisible(bool)
  getElement()

src/obsidian/JsonFileView.ts       Integration
  - loadCompanionSchema() async  - read sibling .schema.json via vault
  - setSchema(text)              - public for tests + manual injection
  - currentSchema, currentErrors
  - applyValidation()            - run validate, push to UI
  passes errors to TreeView.setValidationErrors(...)

src/obsidian/TreeView.ts
  setValidationErrors(map: Map<pathStr, message>)
  - on render: add .json-row-error CSS class + title-attr on matching rows

src/obsidian/SettingsTab.ts        +2 settings (master switch + suffix)
```

## 3. Core Module (`src/core/schema.ts`)

```ts
import Ajv from "ajv";
import type { JsonValue, JsonPath } from "./types";

export interface PathError {
  path: JsonPath;
  message: string;
}

export interface CompiledSchema {
  validate: (value: JsonValue) => PathError[];
}

export type SchemaCompileResult =
  | { ok: true; schema: CompiledSchema }
  | { ok: false; error: string };

export function compileSchema(text: string): SchemaCompileResult { … }
```

**Pfad-Mapping aus Ajv-Errors:**

Ajv liefert `instancePath` als JSON-Pointer (`""`, `"/foo"`, `"/arr/2"`). Wir konvertieren zu `JsonPath` (`[]`, `["foo"]`, `["arr", 2]`).

**Strict mode aus:** Wir kompilieren mit `{ strict: false, allErrors: true }` damit
- nicht-standard `$schema`-URLs nicht crashen
- ALLE Fehler zurückkommen, nicht nur der erste

## 4. Companion-File-Pattern

Bei `setViewData(text, _clear)`:

```ts
if (this.settings.validateAgainstSchema && this.file) {
  const schemaPath = this.file.path.replace(/\.json$/, this.settings.companionSchemaSuffix);
  const sibling = this.app.vault.getAbstractFileByPath(schemaPath);
  if (sibling instanceof TFile) {
    const schemaText = await this.app.vault.cachedRead(sibling);
    const result = compileSchema(schemaText);
    if (result.ok) this.currentSchema = result.schema;
    else this.showSchemaError(result.error);
  }
}
```

`this.file` ist `null` in einigen Test-Setups → defensiv handeln (kein Schema-Lookup, kein Crash).

## 5. Validation Lifecycle

1. `setViewData` → parse → applyValidation
2. `applyMutation` → already updates `currentValue`+`data` → applyValidation
3. `handleSourceChange` → parse → applyValidation
4. Schema-Setter (manuell oder Companion-Load) → applyValidation

`applyValidation()`:
- Bei `currentValue === null` (Parse-Fehler): clear errors, banner ausblenden.
- Sonst: `currentSchema?.validate(currentValue)` → errors → setze auf Banner + TreeView (falls aktiv) + memorize für späteren Render.

## 6. UI

### 6.1 Schema-Banner

Komponente analog zu `.json-error-banner`. Position direkt darüber. Klassen-Variante:
- `.json-schema-banner` für 1+ Fehler (orange/red accent)
- Hidden wenn 0 Fehler
- Text: `"3 schema errors — hover red rows for details"` (oder `"1 schema error — …"`)

### 6.2 Row-Error-Marker

In `TreeView.attachStructuralActions` oder `attachCopyButtons` ergänzen: bei jeder Row, falls Pfad in `validationErrors` → Klasse `.json-row-error` + `title=` mit Message setzen.

CSS:
```css
.json-row.json-row-error {
  outline: 1px solid var(--text-error, #d32f2f);
  outline-offset: -1px;
  border-radius: var(--jv-radius-sm);
}
.json-row.json-row-error > .json-key {
  color: var(--text-error, #d32f2f);
}
```

Tooltip (Obsidian's native title attr für Phase 1 — Hover-Tooltip-Komponente wäre nicer, aber das Tooltip-Modul ist auf andere Use-Cases gemünzt).

### 6.3 Settings

- `validateAgainstSchema: boolean` (default `true`)
- `companionSchemaSuffix: string` (default `.schema.json`)

## 8. Risiken

- **Ajv-Bundle-Size**: ajv@8 ist ~150 KB minified. Trade-off akzeptabel für die Funktionalität. esbuild bundlet inline.
- **Performance**: Ajv compile ist langsam (~50ms). Wir cachen die compiled fn pro Datei. Re-compile nur bei Schema-Text-Change.
- **Tests ohne Vault**: Companion-File-Lookup geht nicht in happy-dom (kein File-System). Tests benutzen `setSchema(text)` direkt.
