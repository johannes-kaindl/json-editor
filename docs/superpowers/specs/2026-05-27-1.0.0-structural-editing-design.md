# 1.0.0 — Structural Editing — Design-Spec

**Projekt:** `obsidian-json-editor`
**Datum:** 2026-05-27
**Status:** Spec, freigegeben zur Umsetzung (autonomer Run)
**Entstanden aus:** AGENTS.md Backlog §v2.0 + autonomes Roadmap-Sweep nach 0.3.0

Bisher konnte der Tree-View nur **Werte** ändern. 1.0.0 macht ihn zum vollwertigen JSON-Editor: **Struktur** (Keys hinzufügen / löschen / umbenennen, Array-Items hinzufügen / löschen) wird direkt im Tree-Mode editierbar — plus **Tree-Mode-Undo/Redo** für alle Edits.

---

## 1. Scope-Entscheidung & Decomposition

Das ursprüngliche Roadmap-Item "1.0.0 — Structural editing" listete fünf große Bereiche. Realistisch ist das zwei+ Tage Arbeit und versteckt unterschiedliche Komplexitäts-Profile. Ich teile auf:

**1.0.0 (diese Spec) — der Kern-Tree-Editor:**
- Add Key (Objects), Add Item (Arrays)
- Delete Row
- Rename Key (Objects)
- Tree-Mode Undo/Redo

**1.1.0 (separate Spec, später) — Polish:**
- Type-Switching für Primitives (string ↔ number ↔ boolean ↔ null ↔ object ↔ array)
- Drag-and-Drop Reorder

**1.2.0 (separate Spec, später) — Source/Tree-Brücke:**
- Cross-Mode Unified Undo/Redo (CodeMirror-Stack + Tree-Stack vereinen)

**1.3.0 (optional, separate Spec):**
- JSON Schema Validation via `ajv`

Begründung: 1.0.0 muss ein klar abgrenzbarer Milestone sein. Wenn der Tree CRUD und Undo kann, ist er ein "echter Editor". Type-Switch und Drag-Drop sind Polish, Cross-Mode-Undo ist eigene Komplexität (CodeMirror-Integration), JSON Schema ist ganz eigenes System.

Wenn der User das anders sieht, kann er die nachfolgenden Releases umordnen.

---

## 2. Design-Prinzipien

1. **Editor-Symmetrie.** Was du in Source kannst, kannst du jetzt auch in Tree.
2. **Discoverable UI.** Hover-Affordances mit klaren Icons. Keyboard-Shortcuts dokumentiert.
3. **Immutable core ops.** Alle Mutationen liegen in `src/core/` als pure `(value, path, …) → JsonValue` Funktionen. UI ist dünner Adapter.
4. **Undo first-class.** Jede Mutation pushed Pre-State auf Stack. Keine Spezialfälle.
5. **Keine destruktiven Aktionen ohne Vorbereitung.** Delete ist instant (gefolgt von Undo-Möglichkeit), nicht confirmation-Dialog (das nervt im Editor-Flow).
6. **Validierung minimal aber korrekt.** Empty Keys verboten. Duplicate Keys verboten (überschreiben wäre Datenverlust). Andere Constraints kommen mit 1.3.0 (Schema).

## 3. Core Operations

Neue Funktionen in `src/core/edit.ts`:

```ts
export function addObjectKey(value: JsonValue, parentPath: JsonPath, key: string, newVal: JsonValue): JsonValue;
export function addArrayItem(value: JsonValue, parentPath: JsonPath, newVal: JsonValue, atIndex?: number): JsonValue;
export function deleteAt(value: JsonValue, path: JsonPath): JsonValue;
export function renameKey(value: JsonValue, path: JsonPath, newKey: string): JsonValue;
```

**Validierung:**
- `addObjectKey`: throws if `key === ""` or key already exists in parent.
- `renameKey`: throws if `newKey === ""`, `newKey === currentKey` (no-op), or `newKey` already exists in parent.
- `deleteAt`: no-op for non-existent path; throws on empty path (can't delete root).
- `addArrayItem`: appends to end if `atIndex` omitted; otherwise splice in.

Errors sind plain `Error` mit klarer Message; UI fängt sie und zeigt Notice/Toast.

## 4. UI Affordances

### 4.1 Row-Actions (hover)

Jede `.json-row` bekommt im Hover-State eine kleine Action-Group (rechts neben den Copy-Buttons). Drei Buttons:

| Button | Icon | Action |
|---|---|---|
| Rename key | pencil | objects only — Inline-Edit auf dem Key-Span |
| Delete | trash | Delete this row |
| (no "Add" here — Add ist Container-Affordance) | | |

Tooltip per button (`title` attr): "Rename key", "Delete row".

### 4.2 Container-Actions (always visible)

Jeder `.json-container` bekommt eine "+ Add" Affordance unten in seiner content-Section, nach allen rows.

```
{
  "a": 1,
  "b": 2,
  + Add key             ← clickable
}
```

Für Arrays:
```
[
  1, 2, 3,
  + Add item             ← clickable
]
```

Click on "+ Add key" (object): replaces the chip with an inline input (key-name). Enter commits + adds key with default value `null`. Esc cancels.

Click on "+ Add item" (array): appends new item with default value `null` and starts inline-edit on its value (or just adds null and lets user click to edit).

### 4.3 Keyboard Shortcuts

| Shortcut | Action |
|---|---|
| `Backspace` / `Delete` on focused row | Delete that row (uses undo stack) |
| `Cmd/Ctrl + Z` in tree mode | Undo last tree mutation |
| `Cmd/Ctrl + Shift + Z` in tree mode | Redo |
| `Cmd/Ctrl + N` ? | NO — not adding a new global shortcut. Add-via-button only. |

The existing keyboard nav from 0.3.0 stays. Backspace/Delete is new.

## 5. Undo / Redo

### 5.1 Architecture

`src/core/history.ts` (new):

```ts
export interface HistoryState {
  value: JsonValue;
  description: string;  // e.g. "Add key 'foo'", for tooltips later
}

export class History {
  private undo: HistoryState[] = [];
  private redo: HistoryState[] = [];

  push(state: HistoryState): void;
  undo(): HistoryState | null;
  redo(): HistoryState | null;
  canUndo(): boolean;
  canRedo(): boolean;
  clear(): void;
}
```

- Capacity: 100 states. Older trimmed FIFO.
- On any user-initiated mutation (value-edit, add, del, rename), JsonFileView pushes the PRE-mutation state to undo stack, applies mutation, clears redo stack.
- On undo: pop undo, push current state to redo, set current to popped.
- On redo: pop redo, push current state to undo, set current to popped.
- Switching to source mode clears the history (source has its own undo). Switching back from source starts fresh.

### 5.2 Trigger Points

```
User does edit → mutation function returns newValue
                  ↓
            JsonFileView.applyMutation(newValue, description)
                  ↓
            history.push(currentState)
            history.redo.clear()
            currentValue = newValue
            serialize + requestSave
            re-render tree
            re-apply filter
```

Keyboard handler in JsonFileView for Cmd+Z / Cmd+Shift+Z:
- Tree mode only. In source mode, CodeMirror handles its own.
- Registered via Obsidian's addCommand with checkCallback.

## 7. Edge Cases

| Fall | Verhalten |
|---|---|
| Add key with name same as existing | Show Obsidian Notice "Key already exists"; don't add; keep input open |
| Add key with empty name | Show Notice "Key cannot be empty"; don't add |
| Rename to existing key | Same — Notice + don't apply |
| Delete the root container | Can't — the row buttons exist on rows only, not on the root container. Backspace on a root-level primitive value is undefined behavior — skip (no row to delete). |
| Undo past history start | No-op; gracefully handled by `canUndo()` |
| Redo after a new mutation | Redo stack cleared on new mutation |
| Edit in source after some tree edits | History clears on mode switch |
| Filter active + delete a matching row | Row vanishes; filter re-applies; match count updates |
| Add to a collapsed container | Container auto-expands first |
