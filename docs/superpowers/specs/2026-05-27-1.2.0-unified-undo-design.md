# 1.2.0 — Cross-mode unified Undo/Redo — Design-Spec

**Projekt:** `obsidian-json-editor`
**Datum:** 2026-05-27
**Status:** Spec, freigegeben (autonomer Run nach 1.1.0)
**Entstanden aus:** Backlog-Item §3 in `AGENTS.md` / 1.0.0-Scope-Decomposition.

In 1.0.0 hat Tree-Mode einen eigenen Undo-Stack; Mode-Switch leert ihn. Source-Mode hat CodeMirrors lokale Undo-History. Beide existieren parallel und sind sich nicht bewusst. 1.2.0 ersetzt beide durch eine **einheitliche, text-basierte History** die Mode-Switches überlebt.

---

## 1. Scope

**IN:**
- `History<T>` wird generisch (statt fester `HistoryState`)
- Ein einziger `History<string>` Stack in `JsonFileView`
- Tree-Mutationen pushen Pre-State-Text (statt JsonValue-Snapshot)
- Source-Mutationen pushen Pre-State-Text (vor jedem onChange)
- Mode-Switch leert NICHT
- Cmd/Ctrl+Z funktioniert in beiden Modi und über Mode-Switches hinweg
- Redo (Cmd/Ctrl+Shift+Z) ditto

**OUT:**
- Persistente History über Dateischluss/-öffnen hinweg (Reset auf File-Load)
- Group-by-time-window batching (jede Änderung = ein Eintrag)
- Source-Mode behält seine eigene CodeMirror-History — wir ersetzen sie

---

## 2. Design-Entscheidungen

1. **Text-Snapshot statt JsonValue-Snapshot.**
   Tree-Mutationen serialisieren ohnehin nach jedem Edit; der Text ist die kanonische Repräsentation. Vorteil: Source-Mode-Snapshots und Tree-Mode-Snapshots haben dieselbe Form → eine Datenstruktur.

2. **CodeMirror's lokale History entfernen.**
   Wenn unsere unified history pro `onChange` pushed, ist sie de facto identisch granular wie CodeMirrors lokale Tastenanschlag-History. Beides parallel wäre konfusionerzeugend (zwei Stacks die unterschiedlich tief sind).

3. **Mode-Switch erhält die History.**
   Das ist DER Punkt der Spec. Vorher: Switch → tree-stack.clear(). Jetzt: Switch passiert ohne History-Effekt.

4. **Undo restored Text + erkennt Mode automatisch.**
   `undo()` setzt nur den Text auf den Pre-State. Wenn der Pre-State zur damaligen Mode-Konfiguration in Tree-Mode interpretierbar (parsbar) war, bleibt Tree-Mode aktiv. Sonst: Source-Mode (kein Mode-Switch implicit — die History speichert KEINEN Mode, nur Text).

   Konkreter: Wir setzen einfach `this.data = prevText` und `refreshMode()`. Wenn Tree-Mode aktiv ist und der Text parsbar ist, klappt das. Wenn er invalid wird, force-switch zu Source mit Banner (existierende `setViewData` Logik).

5. **History-Cap auf 100 bleibt** (1.0.0-Design). Erste Schicht fällt heraus bei Überlauf.

---

## 3. Core (`src/core/history.ts`)

Refactor von `History` (concrete) auf `History<T>` (generic). Erhalten:
- `push(state: T): void`
- `undo(currentState: T): T | null`
- `redo(currentState: T): T | null`
- `canUndo(): boolean`
- `canRedo(): boolean`
- `clear(): void`

Bestehende `HistoryState`-Konsumenten in JsonFileView nutzen ab jetzt `History<string>`. `description` fällt weg (war Debug-Bonus; in Praxis ungenutzt).

---

## 4. JsonFileView-Änderungen

```ts
// vorher:
private history = new History();   // History<HistoryState>
// nachher:
private history = new History<string>();
```

`applyMutation(newValue)`:
- Vor dem Mutate: `history.push(this.data)` (alter Text)
- Wie bisher: `this.data = serialize(newValue, ...)` + `requestSave()`

`handleSourceChange(text)`:
- Vorher: nur parse + state-update
- Jetzt: `history.push(this.data)` (alter Text) + dann state-update.

`switchTo(target)`:
- Entfernen: `this.history.clear()`

`undo()` / `redo()`:
- `prev = history.undo(this.data)` / `next = history.redo(this.data)`
- `setViewData(prev, false)` (geht durch parse + refreshMode automatisch)

`canUndo()` / `canRedo()`:
- `return this.history.canUndo()` — NICHT mehr mode-gated.

## 8. Risiken

- **CodeMirror-history removal**: Charakter-undo geht in source mode an unsere `onChange` → ein push pro keystroke. Das ist gleich granular wie CM. Cap 100 = 100 keystrokes undo. Reicht.
- **Performance**: Jeder String-Push klont den String (V8 macht das via reference + COW, also billig). 100 strings à durchschnittlich 1-50 KB → max ~5 MB Memory, vernachlässigbar.
- **External save during undo loop**: `requestSave()` wird mehrfach getriggert; Obsidian batcht. OK.

```markdown
## [1.2.0] — 2026-05-27

### Changed
- **Unified cross-mode undo/redo.** Tree-mode and source-mode now share a single 100-deep text-based history. Cmd/Ctrl+Z and Cmd/Ctrl+Shift+Z work in both modes and across mode switches.
- Switching mode no longer clears the undo history.
- Source mode no longer keeps a separate CodeMirror-local undo stack; the unified history captures every doc change.

### Internal
- `History` is now generic (`History<T>`); JsonFileView uses `History<string>` instead of the previous tree-only `History<HistoryState>`.
```
